/AGENTS.md:
--------------------------------------------------------------------------------
 1 | # Ralph Agent Instructions
 2 | 
 3 | ## Overview
 4 | 
 5 | Ralph is an autonomous AI agent loop that runs Amp repeatedly until all PRD items are complete. Each iteration is a fresh Amp instance with clean context.
 6 | 
 7 | ## Commands
 8 | 
 9 | ```bash
10 | # Run the flowchart dev server
11 | cd flowchart && npm run dev
12 | 
13 | # Build the flowchart
14 | cd flowchart && npm run build
15 | 
16 | # Run Ralph (from your project that has prd.json)
17 | ./ralph.sh [max_iterations]
18 | ```
19 | 
20 | ## Key Files
21 | 
22 | - `ralph.sh` - The bash loop that spawns fresh Amp instances
23 | - `prompt.md` - Instructions given to each Amp instance
24 | - `prd.json.example` - Example PRD format
25 | - `flowchart/` - Interactive React Flow diagram explaining how Ralph works
26 | 
27 | ## Flowchart
28 | 
29 | The `flowchart/` directory contains an interactive visualization built with React Flow. It's designed for presentations - click through to reveal each step with animations.
30 | 
31 | To run locally:
32 | ```bash
33 | cd flowchart
34 | npm install
35 | npm run dev
36 | ```
37 | 
38 | ## Patterns
39 | 
40 | - Each iteration spawns a fresh Amp instance with clean context
41 | - Memory persists via git history, `progress.txt`, and `prd.json`
42 | - Stories should be small enough to complete in one context window
43 | - Always update AGENTS.md with discovered patterns for future iterations
44 |

/ralph.sh:
--------------------------------------------------------------------------------
 1 | #!/bin/bash
 2 | # Ralph Wiggum - Long-running AI agent loop
 3 | # Usage: ./ralph.sh [max_iterations]
 4 | 
 5 | set -e
 6 | 
 7 | MAX_ITERATIONS=${1:-10}
 8 | SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
 9 | PRD_FILE="$SCRIPT_DIR/prd.json"
10 | PROGRESS_FILE="$SCRIPT_DIR/progress.txt"
11 | ARCHIVE_DIR="$SCRIPT_DIR/archive"
12 | LAST_BRANCH_FILE="$SCRIPT_DIR/.last-branch"
13 | 
14 | # Archive previous run if branch changed
15 | if [ -f "$PRD_FILE" ] && [ -f "$LAST_BRANCH_FILE" ]; then
16 |   CURRENT_BRANCH=$(jq -r '.branchName // empty' "$PRD_FILE" 2>/dev/null || echo "")
17 |   LAST_BRANCH=$(cat "$LAST_BRANCH_FILE" 2>/dev/null || echo "")
18 |   
19 |   if [ -n "$CURRENT_BRANCH" ] && [ -n "$LAST_BRANCH" ] && [ "$CURRENT_BRANCH" != "$LAST_BRANCH" ]; then
20 |     # Archive the previous run
21 |     DATE=$(date +%Y-%m-%d)
22 |     # Strip "ralph/" prefix from branch name for folder
23 |     FOLDER_NAME=$(echo "$LAST_BRANCH" | sed 's|^ralph/||')
24 |     ARCHIVE_FOLDER="$ARCHIVE_DIR/$DATE-$FOLDER_NAME"
25 |     
26 |     echo "Archiving previous run: $LAST_BRANCH"
27 |     mkdir -p "$ARCHIVE_FOLDER"
28 |     [ -f "$PRD_FILE" ] && cp "$PRD_FILE" "$ARCHIVE_FOLDER/"
29 |     [ -f "$PROGRESS_FILE" ] && cp "$PROGRESS_FILE" "$ARCHIVE_FOLDER/"
30 |     echo "   Archived to: $ARCHIVE_FOLDER"
31 |     
32 |     # Reset progress file for new run
33 |     echo "# Ralph Progress Log" > "$PROGRESS_FILE"
34 |     echo "Started: $(date)" >> "$PROGRESS_FILE"
35 |     echo "---" >> "$PROGRESS_FILE"
36 |   fi
37 | fi
38 | 
39 | # Track current branch
40 | if [ -f "$PRD_FILE" ]; then
41 |   CURRENT_BRANCH=$(jq -r '.branchName // empty' "$PRD_FILE" 2>/dev/null || echo "")
42 |   if [ -n "$CURRENT_BRANCH" ]; then
43 |     echo "$CURRENT_BRANCH" > "$LAST_BRANCH_FILE"
44 |   fi
45 | fi
46 | 
47 | # Initialize progress file if it doesn't exist
48 | if [ ! -f "$PROGRESS_FILE" ]; then
49 |   echo "# Ralph Progress Log" > "$PROGRESS_FILE"
50 |   echo "Started: $(date)" >> "$PROGRESS_FILE"
51 |   echo "---" >> "$PROGRESS_FILE"
52 | fi
53 | 
54 | echo "Starting Ralph - Max iterations: $MAX_ITERATIONS"
55 | 
56 | for i in $(seq 1 $MAX_ITERATIONS); do
57 |   echo ""
58 |   echo "═══════════════════════════════════════════════════════"
59 |   echo "  Ralph Iteration $i of $MAX_ITERATIONS"
60 |   echo "═══════════════════════════════════════════════════════"
61 |   
62 |   # Run amp with the ralph prompt
63 |   OUTPUT=$(cat "$SCRIPT_DIR/prompt.md" | amp --dangerously-allow-all 2>&1 | tee /dev/stderr) || true
64 |   
65 |   # Check for completion signal
66 |   if echo "$OUTPUT" | grep -q "<promise>COMPLETE</promise>"; then
67 |     echo ""
68 |     echo "Ralph completed all tasks!"
69 |     echo "Completed at iteration $i of $MAX_ITERATIONS"
70 |     exit 0
71 |   fi
72 |   
73 |   echo "Iteration $i complete. Continuing..."
74 |   sleep 2
75 | done
76 | 
77 | echo ""
78 | echo "Ralph reached max iterations ($MAX_ITERATIONS) without completing all tasks."
79 | echo "Check $PROGRESS_FILE for status."
80 | exit 1
81 |

  1 | # Ralph Agent Instructions
  2 | 
  3 | You are an autonomous coding agent working on a software project.
  4 | 
  5 | ## Your Task
  6 | 
  7 | 1. Read the PRD at `prd.json` (in the same directory as this file)
  8 | 2. Read the progress log at `progress.txt` (check Codebase Patterns section first)
  9 | 3. Check you're on the correct branch from PRD `branchName`. If not, check it out or create from main.
 10 | 4. Pick the **highest priority** user story where `passes: false`
 11 | 5. Implement that single user story
 12 | 6. Run quality checks (e.g., typecheck, lint, test - use whatever your project requires)
 13 | 7. Update AGENTS.md files if you discover reusable patterns (see below)
 14 | 8. If checks pass, commit ALL changes with message: `feat: [Story ID] - [Story Title]`
 15 | 9. Update the PRD to set `passes: true` for the completed story
 16 | 10. Append your progress to `progress.txt`
 17 | 
 18 | ## Progress Report Format
 19 | 
 20 | APPEND to progress.txt (never replace, always append):
 21 | ```
 22 | ## [Date/Time] - [Story ID]
 23 | Thread: https://ampcode.com/threads/$AMP_CURRENT_THREAD_ID
 24 | - What was implemented
 25 | - Files changed
 26 | - **Learnings for future iterations:**
 27 |   - Patterns discovered (e.g., "this codebase uses X for Y")
 28 |   - Gotchas encountered (e.g., "don't forget to update Z when changing W")
 29 |   - Useful context (e.g., "the evaluation panel is in component X")
 30 | ---
 31 | ```
 32 | 
 33 | Include the thread URL so future iterations can use the `read_thread` tool to reference previous work if needed.
 34 | 
 35 | The learnings section is critical - it helps future iterations avoid repeating mistakes and understand the codebase better.
 36 | 
 37 | ## Consolidate Patterns
 38 | 
 39 | If you discover a **reusable pattern** that future iterations should know, add it to the `## Codebase Patterns` section at the TOP of progress.txt (create it if it doesn't exist). This section should consolidate the most important learnings:
 40 | 
 41 | ```
 42 | ## Codebase Patterns
 43 | - Example: Use `sql<number>` template for aggregations
 44 | - Example: Always use `IF NOT EXISTS` for migrations
 45 | - Example: Export types from actions.ts for UI components
 46 | ```
 47 | 
 48 | Only add patterns that are **general and reusable**, not story-specific details.
 49 | 
 50 | ## Update AGENTS.md Files
 51 | 
 52 | Before committing, check if any edited files have learnings worth preserving in nearby AGENTS.md files:
 53 | 
 54 | 1. **Identify directories with edited files** - Look at which directories you modified
 55 | 2. **Check for existing AGENTS.md** - Look for AGENTS.md in those directories or parent directories
 56 | 3. **Add valuable learnings** - If you discovered something future developers/agents should know:
 57 |    - API patterns or conventions specific to that module
 58 |    - Gotchas or non-obvious requirements
 59 |    - Dependencies between files
 60 |    - Testing approaches for that area
 61 |    - Configuration or environment requirements
 62 | 
 63 | **Examples of good AGENTS.md additions:**
 64 | - "When modifying X, also update Y to keep them in sync"
 65 | - "This module uses pattern Z for all API calls"
 66 | - "Tests require the dev server running on PORT 3000"
 67 | - "Field names must match the template exactly"
 68 | 
 69 | **Do NOT add:**
 70 | - Story-specific implementation details
 71 | - Temporary debugging notes
 72 | - Information already in progress.txt
 73 | 
 74 | Only update AGENTS.md if you have **genuinely reusable knowledge** that would help future work in that directory.
 75 | 
 76 | ## Quality Requirements
 77 | 
 78 | - ALL commits must pass your project's quality checks (typecheck, lint, test)
 79 | - Do NOT commit broken code
 80 | - Keep changes focused and minimal
 81 | - Follow existing code patterns
 82 | 
 83 | ## Browser Testing (Required for Frontend Stories)
 84 | 
 85 | For any story that changes UI, you MUST verify it works in the browser:
 86 | 
 87 | 1. Load the `dev-browser` skill
 88 | 2. Navigate to the relevant page
 89 | 3. Verify the UI changes work as expected
 90 | 4. Take a screenshot if helpful for the progress log
 91 | 
 92 | A frontend story is NOT complete until browser verification passes.
 93 | 
 94 | ## Stop Condition
 95 | 
 96 | After completing a user story, check if ALL stories have `passes: true`.
 97 | 
 98 | If ALL stories are complete and passing, reply with:
 99 | <promise>COMPLETE</promise>
100 | 
101 | If there are still stories with `passes: false`, end your response normally (another iteration will pick up the next story).
102 | 
103 | ## Important
104 | 
105 | - Work on ONE story per iteration
106 | - Commit frequently
107 | - Keep CI green
108 | - Read the Codebase Patterns section in progress.txt before starting

  1 | # Ralph
  2 | 
  3 | ![Ralph](ralph.webp)
  4 | 
  5 | Ralph is an autonomous AI agent loop that runs [Amp](https://ampcode.com) repeatedly until all PRD items are complete. Each iteration is a fresh Amp instance with clean context. Memory persists via git history, `progress.txt`, and `prd.json`.
  6 | 
  7 | Based on [Geoffrey Huntley's Ralph pattern](https://ghuntley.com/ralph/).
  8 | 
  9 | [Read my in-depth article on how I use Ralph](https://x.com/ryancarson/status/2008548371712135632)
 10 | 
 11 | ## Prerequisites
 12 | 
 13 | - [Amp CLI](https://ampcode.com) installed and authenticated
 14 | - `jq` installed (`brew install jq` on macOS)
 15 | - A git repository for your project
 16 | 
 17 | ## Setup
 18 | 
 19 | ### Option 1: Copy to your project
 20 | 
 21 | Copy the ralph files into your project:
 22 | 
 23 | ```bash
 24 | # From your project root
 25 | mkdir -p scripts/ralph
 26 | cp /path/to/ralph/ralph.sh scripts/ralph/
 27 | cp /path/to/ralph/prompt.md scripts/ralph/
 28 | chmod +x scripts/ralph/ralph.sh
 29 | ```
 30 | 
 31 | ### Option 2: Install skills globally
 32 | 
 33 | Copy the skills to your Amp config for use across all projects:
 34 | 
 35 | ```bash
 36 | cp -r skills/prd ~/.config/amp/skills/
 37 | cp -r skills/ralph ~/.config/amp/skills/
 38 | ```
 39 | 
 40 | ### Configure Amp auto-handoff (recommended)
 41 | 
 42 | Add to `~/.config/amp/settings.json`:
 43 | 
 44 | ```json
 45 | {
 46 |   "amp.experimental.autoHandoff": { "context": 90 }
 47 | }
 48 | ```
 49 | 
 50 | This enables automatic handoff when context fills up, allowing Ralph to handle large stories that exceed a single context window.
 51 | 
 52 | ## Workflow
 53 | 
 54 | ### 1. Create a PRD
 55 | 
 56 | Use the PRD skill to generate a detailed requirements document:
 57 | 
 58 | ```
 59 | Load the prd skill and create a PRD for [your feature description]
 60 | ```
 61 | 
 62 | Answer the clarifying questions. The skill saves output to `tasks/prd-[feature-name].md`.
 63 | 
 64 | ### 2. Convert PRD to Ralph format
 65 | 
 66 | Use the Ralph skill to convert the markdown PRD to JSON:
 67 | 
 68 | ```
 69 | Load the ralph skill and convert tasks/prd-[feature-name].md to prd.json
 70 | ```
 71 | 
 72 | This creates `prd.json` with user stories structured for autonomous execution.
 73 | 
 74 | ### 3. Run Ralph
 75 | 
 76 | ```bash
 77 | ./scripts/ralph/ralph.sh [max_iterations]
 78 | ```
 79 | 
 80 | Default is 10 iterations.
 81 | 
 82 | Ralph will:
 83 | 1. Create a feature branch (from PRD `branchName`)
 84 | 2. Pick the highest priority story where `passes: false`
 85 | 3. Implement that single story
 86 | 4. Run quality checks (typecheck, tests)
 87 | 5. Commit if checks pass
 88 | 6. Update `prd.json` to mark story as `passes: true`
 89 | 7. Append learnings to `progress.txt`
 90 | 8. Repeat until all stories pass or max iterations reached
 91 | 
 92 | ## Key Files
 93 | 
 94 | | File | Purpose |
 95 | |------|---------|
 96 | | `ralph.sh` | The bash loop that spawns fresh Amp instances |
 97 | | `prompt.md` | Instructions given to each Amp instance |
 98 | | `prd.json` | User stories with `passes` status (the task list) |
 99 | | `prd.json.example` | Example PRD format for reference |
100 | | `progress.txt` | Append-only learnings for future iterations |
101 | | `skills/prd/` | Skill for generating PRDs |
102 | | `skills/ralph/` | Skill for converting PRDs to JSON |
103 | | `flowchart/` | Interactive visualization of how Ralph works |
104 |

106 | 
107 | [![Ralph Flowchart](ralph-flowchart.png)](https://snarktank.github.io/ralph/)
108 | 
109 | **[View Interactive Flowchart](https://snarktank.github.io/ralph/)** - Click through to see each step with animations.
110 | 
111 | The `flowchart/` directory contains the source code. To run locally:
112 | 
113 | ```bash
114 | cd flowchart
115 | npm install
116 | npm run dev
117 | ```

119 | ## Critical Concepts
120 | 
121 | ### Each Iteration = Fresh Context
122 | 
123 | Each iteration spawns a **new Amp instance** with clean context. The only memory between iterations is:
124 | - Git history (commits from previous iterations)
125 | - `progress.txt` (learnings and context)
126 | - `prd.json` (which stories are done)
127 | 
128 | ### Small Tasks
129 | 
130 | Each PRD item should be small enough to complete in one context window. If a task is too big, the LLM runs out of context before finishing and produces poor code.
131 | 
132 | Right-sized stories:
133 | - Add a database column and migration
134 | - Add a UI component to an existing page
135 | - Update a server action with new logic
136 | - Add a filter dropdown to a list
137 | 
138 | Too big (split these):
139 | - "Build the entire dashboard"
140 | - "Add authentication"
141 | - "Refactor the API"
142 | 
143 | ### AGENTS.md Updates Are Critical
144 | 
145 | After each iteration, Ralph updates the relevant `AGENTS.md` files with learnings. This is key because Amp automatically reads these files, so future iterations (and future human developers) benefit from discovered patterns, gotchas, and conventions.
146 | 
147 | Examples of what to add to AGENTS.md:
148 | - Patterns discovered ("this codebase uses X for Y")
149 | - Gotchas ("do not forget to update Z when changing W")
150 | - Useful context ("the settings panel is in component X")
151 | 
152 | ### Feedback Loops
153 | 
154 | Ralph only works if there are feedback loops:
155 | - Typecheck catches type errors
156 | - Tests verify behavior
157 | - CI must stay green (broken code compounds across iterations)
158 | 
159 | ### Browser Verification for UI Stories
160 | 
161 | Frontend stories must include "Verify in browser using dev-browser skill" in acceptance criteria. Ralph will use the dev-browser skill to navigate to the page, interact with the UI, and confirm changes work.
162 | 
163 | ### Stop Condition
164 | 
165 | When all stories have `passes: true`, Ralph outputs `<promise>COMPLETE</promise>` and the loop exits.
166 | 
167 | ## Debugging
168 | 
169 | Check current state:
170 | 
171 | ```bash
172 | # See which stories are done
173 | cat prd.json | jq '.userStories[] | {id, title, passes}'
174 | 
175 | # See learnings from previous iterations
176 | cat progress.txt
177 | 
178 | # Check git history
179 | git log --oneline -10
180 | ```
181 | 
182 | ## Customizing prompt.md
183 | 
184 | Edit `prompt.md` to customize Ralph's behavior for your project:
185 | - Add project-specific quality check commands
186 | - Include codebase conventions
187 | - Add common gotchas for your stack
188 | 
189 | ## Archiving
190 | 
191 | Ralph automatically archives previous runs when you start a new feature (different `branchName`). Archives are saved to `archive/YYYY-MM-DD-feature-name/`.
192 | 
193 | ## References
194 | 
195 | - [Geoffrey Huntley's Ralph article](https://ghuntley.com/ralph/)
196 | - [Amp documentation](https://ampcode.com/manual)

  2 | name: prd
  3 | description: "Generate a Product Requirements Document (PRD) for a new feature. Use when planning a feature, starting a new project, or when asked to create a PRD. Triggers on: create a prd, write prd for, plan this feature, requirements for, spec out."
  4 | ---
  5 | 
  6 | # PRD Generator
  7 | 
  8 | Create detailed Product Requirements Documents that are clear, actionable, and suitable for implementation.
  9 | 
 10 | ---
 11 | 
 12 | ## The Job
 13 | 
 14 | 1. Receive a feature description from the user
 15 | 2. Ask 3-5 essential clarifying questions (with lettered options)
 16 | 3. Generate a structured PRD based on answers
 17 | 4. Save to `tasks/prd-[feature-name].md`
 18 | 
 19 | **Important:** Do NOT start implementing. Just create the PRD.
 20 | 
 21 | ---
 22 | 
 23 | ## Step 1: Clarifying Questions
 24 | 
 25 | Ask only critical questions where the initial prompt is ambiguous. Focus on:
 26 | 
 27 | - **Problem/Goal:** What problem does this solve?
 28 | - **Core Functionality:** What are the key actions?
 29 | - **Scope/Boundaries:** What should it NOT do?
 30 | - **Success Criteria:** How do we know it's done?
 31 | 
 32 | ### Format Questions Like This:
 33 | 
 34 | ```
 35 | 1. What is the primary goal of this feature?
 36 |    A. Improve user onboarding experience
 37 |    B. Increase user retention
 38 |    C. Reduce support burden
 39 |    D. Other: [please specify]
 40 | 
 41 | 2. Who is the target user?
 42 |    A. New users only
 43 |    B. Existing users only
 44 |    C. All users
 45 |    D. Admin users only
 46 | 
 47 | 3. What is the scope?
 48 |    A. Minimal viable version
 49 |    B. Full-featured implementation
 50 |    C. Just the backend/API
 51 |    D. Just the UI
 52 | ```
 53 | 
 54 | This lets users respond with "1A, 2C, 3B" for quick iteration.
 55 | 
 56 | ---
 57 | 
 58 | ## Step 2: PRD Structure
 59 | 
 60 | Generate the PRD with these sections:
 61 | 
 62 | ### 1. Introduction/Overview
 63 | Brief description of the feature and the problem it solves.
 64 | 
 65 | ### 2. Goals
 66 | Specific, measurable objectives (bullet list).
 67 | 
 68 | ### 3. User Stories
 69 | Each story needs:
 70 | - **Title:** Short descriptive name
 71 | - **Description:** "As a [user], I want [feature] so that [benefit]"
 72 | - **Acceptance Criteria:** Verifiable checklist of what "done" means
 73 | 
 74 | Each story should be small enough to implement in one focused session.
 75 | 
 76 | **Format:**
 77 | ```markdown
 78 | ### US-001: [Title]
 79 | **Description:** As a [user], I want [feature] so that [benefit].
 80 | 
 81 | **Acceptance Criteria:**
 82 | - [ ] Specific verifiable criterion
 83 | - [ ] Another criterion
 84 | - [ ] Typecheck/lint passes
 85 | - [ ] **[UI stories only]** Verify in browser using dev-browser skill
 86 | ```
 87 | 
 88 | **Important:** 
 89 | - Acceptance criteria must be verifiable, not vague. "Works correctly" is bad. "Button shows confirmation dialog before deleting" is good.

 45 | ## Story Size: The Number One Rule
 46 | 
 47 | **Each story must be completable in ONE Ralph iteration (one context window).**
 48 | 
 49 | Ralph spawns a fresh Amp instance per iteration with no memory of previous work. If a story is too big, the LLM runs out of context before finishing and produces broken code.
 50 | 
 51 | ### Right-sized stories:
 52 | - Add a database column and migration
 53 | - Add a UI component to an existing page
 54 | - Update a server action with new logic
 55 | - Add a filter dropdown to a list
 56 | 
 57 | ### Too big (split these):
 58 | - "Build the entire dashboard" - Split into: schema, queries, UI components, filters
 59 | - "Add authentication" - Split into: schema, middleware, login UI, session handling
 60 | - "Refactor the API" - Split into one story per endpoint or pattern
 61 | 
 62 | **Rule of thumb:** If you cannot describe the change in 2-3 sentences, it is too big.
 63 | 
 64 | ---
 65 | 
 66 | ## Story Ordering: Dependencies First
 67 | 
 68 | Stories execute in priority order. Earlier stories must not depend on later ones.
 69 | 
 70 | **Correct order:**
 71 | 1. Schema/database changes (migrations)
 72 | 2. Server actions / backend logic
 73 | 3. UI components that use the backend
 74 | 4. Dashboard/summary views that aggregate data
 75 | 
 76 | **Wrong order:**
 77 | 1. UI component (depends on schema that does not exist yet)
 78 | 2. Schema change
 79 | 
 80 | ---
 81 | 
 82 | ## Acceptance Criteria: Must Be Verifiable
 83 | 
 84 | Each criterion must be something Ralph can CHECK, not something vague.
 85 | 
 86 | ### Good criteria (verifiable):
 87 | - "Add `status` column to tasks table with default 'pending'"
 88 | - "Filter dropdown has options: All, Active, Completed"
 89 | - "Clicking delete shows confirmation dialog"
 90 | - "Typecheck passes"
 91 | - "Tests pass"
 92 | 
 93 | ### Bad criteria (vague):
 94 | - "Works correctly"
 95 | - "User can do X easily"
 96 | - "Good UX"
 97 | - "Handles edge cases"
 98 | 
 99 | ### Always include as final criterion:
100 | ```
101 | "Typecheck passes"
102 | ```
103 | 
104 | For stories with testable logic, also include:
105 | ```
106 | "Tests pass"
107 | ```
108 | 
109 | ### For stories that change UI, also include:
110 | ```
111 | "Verify in browser using dev-browser skill"
112 | ```
113 | 
114 | Frontend stories are NOT complete until visually verified. Ralph will use the dev-browser skill to navigate to the page, interact with the UI, and confirm changes work.
